What is Cross-Site Request Forgery (CSRF)?

<!-- ANSWER -->
Cross-Site Request Forgery (CSRF) is a web security vulnerability where an attacker tricks a victim's browser into sending unintended authenticated requests to a trusted website.

CSRF exploits:
- automatically attached cookies
- authenticated browser sessions
- missing request validation

Example attack:

```html
<form action="https://bank.com/transfer" method="POST">
  <input type="hidden" name="amount" value="10000">
  <input type="hidden" name="to" value="attacker">
</form>

<script>
document.forms[0].submit();
</script>
```

When the victim visits the malicious page:

* the browser automatically sends cookies
* the request becomes authenticated
* the server treats it as legitimate

CSRF mainly affects:

* cookie-based authentication
* session-based applications
* state-changing endpoints

How do CSRF tokens prevent CSRF attacks?

<!-- ANSWER -->
A CSRF token is a random secret value generated by the server and associated with the user's session.

Example form:

```html
<form action="/change-password" method="POST">
  <input type="hidden" name="csrf_token" value="f91ab2">
</form>
```

Server validation:

```js
if (request.body.csrf_token !== session.csrf_token) {
    denyRequest();
}
```

How it works:

| Step            | Explanation                      |
| --------------- | -------------------------------- |
| Token generated | Server creates random token      |
| Token stored    | Saved in user session            |
| Token submitted | Browser sends token with request |
| Validation      | Server compares both values      |

Attackers cannot:

* read the victim's token
* access session data
* generate the correct token

Without the correct token, forged requests are rejected.

How do SameSite cookies help prevent CSRF?

<!-- ANSWER -->
SameSite cookies restrict when browsers send cookies during cross-site requests.

Example:

```http
Set-Cookie: session=abc123; SameSite=Strict
```

Modes:

| SameSite Mode | Behavior                                 |
| ------------- | ---------------------------------------- |
| Strict        | Never send cookies cross-site            |
| Lax           | Send cookies only during safe navigation |
| None          | Always send cookies                      |

How it prevents CSRF:

```text id="q4azde"
Cross-site requests lose authentication cookies.
```

As a result:

* forged requests become unauthenticated
* attackers cannot abuse active sessions
* CSRF attacks fail

Modern browsers apply SameSite protections by default for many cookies.

Why should GET requests never modify server state?

<!-- ANSWER -->
GET requests should only retrieve data.

They should never:
- delete accounts
- transfer money
- change passwords
- modify application state

Dangerous example:

```http
GET /delete-account
```

Attackers can trigger GET requests automatically:

```html
<img src="https://example.com/delete-account">
```

When the victim loads the malicious page:

* the browser sends the GET request automatically
* cookies are attached automatically
* the sensitive action executes

Browsers automatically perform GET requests for:

* images
* scripts
* iframes
* links

Correct REST design:

| Operation   | Recommended Method |
| ----------- | ------------------ |
| Read data   | GET                |
| Create data | POST               |
| Update data | PUT/PATCH          |
| Delete data | DELETE             |

Sensitive operations should always use CSRF protection.

How does Origin validation help prevent CSRF attacks?

<!-- ANSWER -->
The `Origin` header identifies which website initiated the request.

Example request:

```http
Origin: https://example.com
```

Server validation:

```js
if (origin !== "https://example.com") {
    denyRequest();
}
```

Legitimate request:

```http id="2sfcx7"
Origin: https://example.com
```

Forged request:

```http id="1k8qzr"
Origin: https://attacker.com
```

The server rejects requests from untrusted origins.

Benefits:

| Benefit                   | Purpose                     |
| ------------------------- | --------------------------- |
| Simple validation         | Easy to implement           |
| Additional security layer | Detect cross-site requests  |
| Useful for APIs           | Protect sensitive endpoints |

Limitations:

* some requests may omit Origin
* should not replace CSRF tokens entirely

Best practice combines:

* CSRF tokens
* SameSite cookies
* Origin validation

Is JWT authentication vulnerable to CSRF?

<!-- ANSWER -->
It depends on where the JWT is stored.

## JWT in localStorage

```text
Generally NOT vulnerable to CSRF
```

Reason:

* browsers do not automatically attach localStorage values
* attackers cannot force automatic token submission

Requests require explicit JavaScript logic:

```http
Authorization: Bearer <token>
```

---

## JWT in HttpOnly Cookies

```text id="7m2pda"
Vulnerable to CSRF
```

Reason:

* browsers automatically attach cookies
* cross-site requests include authentication credentials
* forged requests become authenticated

Comparison:

| Storage Method   | CSRF Risk |
| ---------------- | --------- |
| localStorage     | Low       |
| Session Cookies  | High      |
| HttpOnly Cookies | High      |

Cookie-based JWT systems should still implement:

* CSRF tokens
* SameSite cookies
* Origin validation

How can XSS vulnerabilities bypass CSRF protections?

<!-- ANSWER -->
XSS (Cross-Site Scripting) allows attackers to execute JavaScript inside the victim's browser.

Once attackers achieve JavaScript execution, they may:
- read CSRF tokens
- submit authenticated requests
- bypass client-side protections
- impersonate users completely

Example:

```js
fetch('/transfer', {
  method: 'POST',
  headers: {
    'X-CSRF-Token': csrfToken
  }
})
```

Why this bypasses CSRF protection:

| Protection        | XSS Impact                           |
| ----------------- | ------------------------------------ |
| CSRF Tokens       | Attacker can read token              |
| SameSite Cookies  | Requests originate from trusted site |
| Origin Validation | Origin appears legitimate            |

Important security principle:

```text id="8z3qma"
XSS can often defeat CSRF protections.
```

Because of this:

* preventing XSS is critical
* output escaping is essential
* CSP should be implemented
* untrusted input must be sanitized

Modern applications must defend against both:

* CSRF
* XSS
